## Part 2: Check-In Schedule ŌĆö checkin_schedule.py (271 lines)

### Data Model

`CheckInEntry` Pydantic model with fields:
- `type_id: CheckInType` ŌĆö Literal of 4 check-in types
- `display_name: str`
- `target_time: time` ŌĆö Python `datetime.time`, user's configured local time
- `staleness_minutes: int` ŌĆö how long after target_time the check-in remains due
- `is_enabled: bool`
- `last_run_date: date | None` ŌĆö the date this last fired

### Design Decisions

**Approach A (config + state in same file)**: Followed the existing TaskStore/MemoryEntryStore pattern where a single JSON file holds both configuration and runtime state. For 4 entries, separate files would be over-engineering.

**Next-due + staleness guard (not time-window)**: A check-in is due when `current_time >= target_time AND current_time <= target_time + staleness_minutes AND last_run_date != today AND is_enabled`. This cannot miss a check-in due to narrow window/poll mismatch. The staleness guard (default 120 min) prevents firing stale check-ins on late bot startup.

**No timezone conversion in this module**: The due-check functions take explicit `current_date` and `current_time` parameters. The caller (heartbeat hook in sub-02) is responsible for converting UTC to the user's configured timezone before calling. This keeps the module pure and testable.
