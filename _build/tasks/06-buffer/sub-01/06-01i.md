### buffer_store.py (248 lines)
Buffer domain model and JSON-persisted store. Contains:
- `Buffer` Pydantic model with 10 fields: id, name, buffer_level, buffer_capacity, recurrence_interval_days, next_due_date (date), alert_threshold, status (active/paused/archived), created_at (datetime), updated_at (datetime)
- `BufferUpdate` partial-update model — excludes buffer_level to force decrement/refill path
- `model_validator(mode="after")` enforcing: level <= capacity, threshold <= capacity
- `Field(ge=...)` constraints: level >= 0, capacity >= 1, interval >= 1, threshold >= 0
- 6 pure helper functions: generate_buffer_id, build_buffer, decrement_buffer, refill_buffer, apply_buffer_updates, serialize/deserialize
- `BufferStore` class: CRUD + decrement + refill + atomic writes (tmp+replace)

## Key Decisions

1. **Dict-keyed store** (matching TaskStore pattern) — O(1) lookups, consistent with project conventions.
2. **Three-status enum** (active/paused/archived) — ADHD-friendly: pause without losing history, archive for completed obligations.
3. **`date` for next_due_date, `datetime` for timestamps** — obligations are date-scoped; audit fields need precision. Matches checkin_schedule.py precedent.
4. **buffer_level excluded from BufferUpdate** — forces all level changes through decrement/refill, protecting the level <= capacity invariant.
5. **model_validator for cross-field constraints** — new to codebase but justified by domain invariants that Field alone can't express.
6. **Rejecting capacity reduction below current level** — safer than silent clamping; user gets explicit error explaining why.
