## Files Created

### voice_trigger_hook.py (136 lines)
5th-position hook in the nanobot-ai chain (after BufferHook). Scans the system prompt for `## Active Check-In:` and `## Buffer Alerts` headings injected by earlier hooks. When triggers are found, voice is enabled via env var, and the cognitive state permits it, injects a `## Voice Delivery` instruction block telling the LLM to use SpeakTool.

Pure functions:
- `is_voice_enabled()` — reads `VOICE_AUTO_ENABLED` env var at each invocation
- `should_auto_voice(state, trigger_type)` — state×trigger matrix (returns bool)
- `detect_checkin_trigger(system_content)` — scans for check-in heading
- `detect_buffer_alert_trigger(system_content)` — scans for buffer alert heading
- `build_voice_delivery_block(voice_checkin, voice_buffer)` — assembles instruction block
- `inject_voice_block_into_prompt(system_content, voice_block)` — appends block to prompt

Hook class:
- `VoiceHook(is_scheduled_session, get_cognitive_state)` — same constructor pattern as SchedulingHook/BufferHook
- `before_iteration(context)` → `_process(context)` with try/except guard

## Key Decisions

1. **Approach B (system prompt instruction)** — followed research recommendation. The hook injects instructions for the LLM to use SpeakTool rather than programmatically calling TTS in a post-response hook. This uses the proven `before_iteration` lifecycle and avoids duplicating the SpeakTool pipeline.

2. **Env var read at each invocation** — `is_voice_enabled()` reads `os.environ` each call rather than caching at construction. Supports runtime toggling without restart.

3. **Trigger detection via heading strings** — detects `## Active Check-In:` and `## Buffer Alerts` as injected by SchedulingHook and BufferHook. These are string constants, not formal contracts. Acceptable coupling given the hook chain is co-designed.

4. **State×trigger matrix** — checkins allowed in baseline+avoidance; buffer alerts only in baseline. All other states suppress auto-voicing. Matches research plan's rationale (don't interrupt flow states, don't add stimulation in overwhelm/rsd).

5. **No markdown stripping** — per research, the SOUL.md instructions tell the LLM to generate speech-appropriate text. No programmatic stripping needed.
