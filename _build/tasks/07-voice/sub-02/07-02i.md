## Files Created

### voice_delivery.py (86 lines)
WAV-to-OGG/Opus conversion module for Telegram voice messages. Three pure/near-pure functions:
- `convert_wav_to_ogg(wav_bytes) -> bytes` — reads WAV from BytesIO, encodes to OGG/Opus via PyAV at 32kbps
- `save_temp_ogg(ogg_bytes) -> Path` — writes OGG bytes to a temp file (`voice_*.ogg`), cleans up on write failure
- `cleanup_temp_file(path) -> None` — removes temp file, logs warning on OSError instead of raising

### voice_tools.py (103 lines)
Nanobot-ai Tool subclass following the established pattern from task_tools.py/memory_tools.py/buffer_tools.py:
- `SpeakTool(Tool)` — accepts text (required), voice (optional str), speed (optional float via NumberSchema with min=0.5/max=3.0)
- `execute()` pipeline: validate text → truncate if >500 chars → synthesize via `asyncio.to_thread(synthesize_speech, ...)` → convert WAV→OGG → save temp file → send via MessageTool.execute(media=[ogg_path]) → cleanup temp file
- `register_voice_tools(registry, message_tool)` — registers SpeakTool with a reference to the MessageTool instance

## Key Decisions

### PyAV for OGG/Opus conversion (Research Approach B)
Chose PyAV over ffmpeg subprocess (Approach A) or bundled binary (Approach C). Rationale: pip-installable with no system dependencies, works on both Mac and Windows, satisfies the "no manual install steps" requirement from the description. The ~50MB download is acceptable given OGG/Opus encoding requires native code.

### SpeakTool wraps MessageTool.execute() (Research Approach 3)
SpeakTool holds a reference to MessageTool and calls `await self._message_tool.execute(content="", media=[ogg_path])`. This avoids the AgentLoop context problem (only tools named "message"/"spawn"/"cron" get routing context injected). By delegating to MessageTool, SpeakTool inherits correct channel/chat_id routing without accessing private attributes.

### asyncio.to_thread for TTS synthesis
`synthesize_speech()` is synchronous and CPU-bound (ONNX inference). Wrapped in `asyncio.to_thread()` to avoid blocking the nanobot event loop during synthesis.

### NumberSchema for speed parameter
Used `NumberSchema(min=0.5, max=3.0)` instead of StringSchema. The nanobot schema system supports numeric types natively, and this gives the LLM proper type hints and range validation.

### Text truncation at 500 characters
Silently truncates text over `MAX_TEXT_LENGTH = 500` rather than rejecting it. Long text produces large audio files and slow synthesis on CPU. 500 chars ≈ 80 words ≈ 30 seconds of speech, which is a reasonable voice message length.

### Temp file with try/finally cleanup
OGG files are saved to disk (required by TelegramChannel which opens with `open(path, "rb")`). Cleanup is in a `finally` block so the temp file is removed even if MessageTool.execute() raises.
