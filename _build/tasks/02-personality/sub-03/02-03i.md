## What Was Built

### 1. SOUL.md State-Aware Adaptation Section (`workspace/SOUL.md`)

Replaced the stub text in `## State-Aware Adaptation` with concrete per-state response rules. Each of the 6 cognitive states has a `### StateName` subsection with 3-6 bullet-point directives:

- **Baseline**: Standard SOUL.md voice, match energy, don't over-scaffold
- **Focus**: Concise responses, don't break momentum, affirm progress briefly
- **Hyperfocus**: Don't interrupt unless critical, gentle water/food/break nudges only
- **Avoidance**: No guilt, ICNU framework, smallest possible first step, externalize difficulty
- **Overwhelm**: Simplify to one single thing, no options/decisions, calming language, feelings first
- **RSD**: Validate emotional experience first, no criticism, frame setbacks as data, no tasks until ready

The section also explains the state indicator mechanism: `[Current cognitive state: STATE_NAME]` is injected by the hook.

### 2. State Response Integration Hook (`state_response_integration.py`, 124 lines)

Implements Approach A from the research plan — static rules in SOUL.md with a lightweight hook that injects a state indicator.

**Exported components:**

- `StateResponseHook` — main hook class. Constructor takes `StateConfig` and `LLMCallable` (dependency injection, consistent with state_detection.py design). Tracks `current_state` in-memory (resets to baseline on restart).
- `HookContext` — Protocol matching nanobot-ai's `AgentHookContext.messages` shape.
- `build_state_indicator(state_name)` — pure function, builds `[Current cognitive state: X]` string.
- `extract_latest_user_message(messages)` — pure function, finds most recent user message from message list.
- `inject_state_into_prompt(system_content, state_name)` — pure function, places/replaces the indicator after `## State-Aware Adaptation` heading in the system prompt.

**Hook lifecycle:**

1. `before_iteration(context)` extracts latest user message
2. Calls `detect_state()` from state_detection.py with the message, current state, config, and LLM callable
3. On success: updates `_current_state`, injects indicator into `context.messages[0]`
4. On any exception: logs warning, falls back to baseline, injects baseline indicator

**Immutability:** The hook replaces `messages[0]` with a spread copy (`{**messages[0], "content": ...}`) rather than mutating the original dict.

## Key Decisions

1. **Approach A (static SOUL.md rules + indicator injection)** — followed research recommendation. All 6 states' rules are always in the prompt (~200 tokens overhead, negligible). Hook stays minimal.

2. **HookContext as Protocol** — avoids importing nanobot-ai internals. Tests use a `MockHookContext` dataclass. The hook works with any object that has a `.messages` list.

3. **In-memory state tracking** — `_current_state` on the hook instance. Resets to baseline on restart, which is correct (you don't know the user's state after a restart). Markov constraints self-correct within 1-2 messages.

4. **No `setup_workspace.py` changes needed** — no new workspace files were added. SOUL.md is already in `TEMPLATE_FILES`.
