## Architecture

Mirrors `task_store.py` exactly in structure:

**Model**: `MemoryEntry(BaseModel)` with `id`, `category` (Literal of 5 types), `content`, `created_at`, `resolved_at` (None = active), `metadata` (dict[str, str]).

**Pure helpers**:
- `generate_entry_id()` тЖТ `uuid.uuid4().hex`
- `build_entry(category, content, metadata)` тЖТ constructs with ID + UTC timestamp
- `resolve_entry_model(entry)` тЖТ returns new entry with `resolved_at` set (immutable)
- `serialize_entries(dict)` / `deserialize_entries(str)` тЖТ JSON round-trip

**Store class**: `MemoryEntryStore` with `__init__(storage_path)`, `_save()` (atomic tmp+rename), `reload()`, `create_entry()`, `get_entry()`, `list_entries()`, `list_active_entries()`, `list_entries_by_category()`, `resolve_entry()`.

## Design Decisions

1. **No UpdateModel** тАФ entries are append-only. Only mutation is `resolve_entry` (soft delete).
2. **`list_entries_by_category` returns active only** тАФ matches primary use case (show active blockers/deadlines). `list_entries()` returns everything if needed.
3. **Resolve is idempotent** тАФ re-resolving updates `resolved_at` timestamp. Simpler than raising on already-resolved.
4. **`metadata: dict[str, str]`** тАФ flexible key-value for category-specific data (due_date, task_id, etc.). Vector embeddings can be added as a separate field later.
5. **No `delete` method** тАФ soft delete via `resolve_entry` is the only removal mechanism, preserving history.
