## Architecture Pattern
Mirrors `task_store.py` exactly: Pydantic model → pure helper functions → Store class with atomic JSON persistence (tmp+rename). Entries are append-only; only mutation is `resolve_entry` (soft delete setting `resolved_at`). `list_entries_by_category` returns active entries only. `list_entries` returns everything.

## Design Decisions
1. No UpdateModel — entries are immutable once created, only resolvable
2. `list_entries_by_category` filters active-only (primary use case for ADHD-critical queries)
3. Resolve is idempotent — re-resolving updates timestamp
4. `metadata: dict[str, str]` — flexible K/V for category-specific data (due_date, task_id). Vector embeddings can be added as separate field later.
5. No hard delete — `resolve_entry` is the only removal mechanism, preserving full history
