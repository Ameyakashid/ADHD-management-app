## What Was Built

**File: `tests/test_bot_smoke.py`** — 18 smoke tests proving the bot config loads into nanobot-ai's runtime and the bot can start.

### Test Classes

| Class | Tests | Requires Credentials | What It Proves |
|---|---|---|---|
| `TestConfigSchemaValidation` | 9 | No | Config template resolves into nanobot-ai's Pydantic `Config` without validation errors. All fields (providers, agents, channels, tools) populate correctly. |
| `TestProviderResolution` | 3 | No | `Config.get_provider()` resolves to openrouter. `get_provider_name()` returns `"openrouter"`. `get_api_key()` returns the configured key. |
| `TestWorkspacePath` | 2 | No | `config.workspace_path` is absolute and under `~/.nanobot/`. |
| `TestToolsConfig` | 2 | No | Web tools enabled, exec tools disabled — matches template. |
| `TestLiveBot` | 2 | Yes (`OPENROUTER_API_KEY`) | `Nanobot.from_config()` succeeds. `bot.run()` returns a response containing `PROOF_OF_LIFE`. |

- Full suite (37 tests including sub-02's) passes with no regressions.

## Key Decisions

1. **Approach C (Hybrid)** from research plan — always-run config validation tests + credential-gated live tests. Best coverage without breaking CI.

2. **Channels are dicts, not typed models.** nanobot-ai stores channel config as `dict` on `ChannelsConfig.telegram`, not a Pydantic model. Tests use dict access (`telegram["enabled"]`) instead of attribute access. This was discovered during test execution and corrected.

3. **No pytest-asyncio dependency.** The live `bot.run()` test uses `asyncio.run()` inside a sync test function instead of adding a `pytest-asyncio` dependency. Follows the code-rules principle of no trivial dependencies.

4. **Fake credentials for always-run tests.** Config validation tests use `FAKE_ENV` with dummy keys. These are sufficient for Pydantic schema validation and provider resolution — nanobot-ai doesn't validate API key format at config-load time.

5. **Workspace override in live tests.** Live tests set `config["agents"]["defaults"]["workspace"]` to a `tmp_path` subdirectory, avoiding writes to the user's real `~/.nanobot/workspace`.

## Deviations from Research

None. Followed Approach C as recommended.
