## Files Created

### task_tools.py (268 lines)
Five nanobot-ai Tool subclasses wrapping TaskStore CRUD operations:

| Tool class | `name` | Delegates to | Read-only |
|---|---|---|---|
| `CreateTaskTool` | `create_task` | `store.create_task()` | No |
| `ListTasksTool` | `list_tasks` | `store.list_tasks()` / `store.list_tasks_by_status()` | Yes |
| `GetTaskTool` | `get_task` | `store.get_task()` | Yes |
| `UpdateTaskTool` | `update_task` | `store.update_task()` | No |
| `CompleteTaskTool` | `complete_task` | `store.mark_complete()` | No |

Plus `register_task_tools(registry, store)` — a single function that registers all five tools into a ToolRegistry.

Three pure helper functions:
- `format_task(task)` — formats a Task into an LLM-readable multi-line string
- `format_task_list(tasks)` — formats a list of tasks, returns "No tasks found." for empty
- `parse_iso_date(value)` — parses ISO 8601 string to timezone-aware datetime, raises ValueError with clear message

## Key Decisions

### Approach A (Tool subclasses) chosen over MCP or CLI
Research identified three approaches. Tool subclasses follow the exact nanobot-ai built-in pattern (ExecTool, ReadFileTool, etc.), give direct access to the TaskStore instance (no IPC overhead), and are testable in isolation. MCP would have added process overhead and file-locking concerns. CLI via exec would have introduced shell injection risk.

### config.json.template not modified
Research confirmed that nanobot-ai v0.1.5's `ToolsConfig` has no mechanism for registering custom Python tools. The `tools` config section only controls built-in tools (web, exec) and MCP servers. Registration is programmatic via `register_task_tools(registry, store)`. Acceptance criterion #4 is satisfied by documenting the registration pattern in the module docstring and providing the registration function.

### Wiring point deferred
The `register_task_tools()` function is ready to call but the actual startup wiring (who calls it and when) is deferred. The most likely integration point is a custom entrypoint that creates the Nanobot instance, then calls `register_task_tools(bot._loop.tools, store)`. This may be addressed in a future integration subtask or in task 03 sub-03.

### `asyncio.run()` for test helper
Used `asyncio.run()` instead of `asyncio.get_event_loop().run_until_complete()` because Python 3.12 raises RuntimeError when no event loop exists in the main thread. The `asyncio.run()` approach creates and closes a loop per call, which is correct for synchronous test execution.

### LLM-friendly parameter schemas
Each tool's JSON schema includes `enum` constraints for status and priority, `nullable` for optional string fields, and descriptive text that tells the LLM what format to use (especially for ISO 8601 dates). This means the LLM gets validation errors from the nanobot framework before `execute()` is even called for type mismatches.

None. The implementation follows Approach A exactly as described in the research report.
